# Design Document: Smart Customer Management Portal

## 1. Problem and Goals
The portal simulates customer operations for a networking hardware company and augments operational views with AI-driven insight modules:
- customer account health scoring
- 90-day churn prediction
- conversational natural language analytics
- weekly account review summary generation

The implementation prioritizes reproducible local demos over production infrastructure.

## 2. Architecture Summary
The application is a single Python codebase built on Streamlit.

### Runtime Components
- `app/main.py`: app shell, sidebar controls, page routing, seed workflow
- `app/db/schema.py`: SQLite schema creation and initialization
- `app/db/repository.py`: data access layer and persistence helpers
- `app/data/generate_synthetic_data.py`: deterministic synthetic data generator
- `app/ai/*.py`: scoring, prediction, NL query, and summary modules
- `app/pages/*.py`: UI pages (CRUD + analytics)

### Data Store
SQLite (`portal.db`) is used for simple setup and deterministic local behavior.

## 3. Data Model
### Core Entities
- `customers`: company profile, plan tier, region, contract dates, NPS, synthetic churn label
- `tickets`: support event stream with severity, status, and resolution time
- `devices`: installed inventory by customer account
- `monthly_usage`: time-series usage/incident metrics per customer/month

### Derived AI Tables
- `health_scores`: latest computed health score and component breakdown
- `churn_predictions`: churn probability, binary flag, and top factors
- `nl_sessions`: conversational context for follow-up query handling
- `generated_emails`: stored weekly summary artifacts

## 4. Synthetic Data Strategy
The synthetic generator seeds 200+ accounts (default 220), with realistic distributions:
- region mix and tier mix weights
- contract duration and renewal timing variability
- monthly usage trends with seasonality and noise
- incident and ticket generation tied to account behavior
- device inventory count tied to plan tier

A deterministic seed enables repeatable demos and test runs.

## 5. AI Integration Points
### 5.1 Account Health Score
A weighted formula produces a 0-100 health score.

Components:
- Ticket quality/risk component (severity + unresolved proportion)
- Contract timing component (renewal proximity)
- Usage stability component (recent trend vs baseline)
- NPS component

The formula is intentionally transparent to support explainability in demos.

### 5.2 Churn Prediction
A supervised classifier (logistic regression) is trained using engineered account features:
- ticket volume and severity
- open/critical ticket counts
- average resolution time
- usage decline percentage
- incident averages
- contract horizon
- plan and region

Model quality is reported with held-out precision, recall, and F1.
Predictions are written back to the database for UI use.

### 5.3 Natural Language Querying
The NL interface uses a safe intent+slot extraction approach in offline mode:
- query intent routing to allowlisted analytical queries
- slot extraction for region/plan filters
- context carry-over via session table for follow-up prompts (for example, “only enterprise plans”)

An optional LLM adapter exists for future enhancement when API keys are configured.

### 5.4 Email Summary Agent
The summary agent composes a weekly narrative per account using:
- latest health score
- latest churn probability
- recent ticket highlights
- usage change signal
- actionable recommendations

Output is generated and stored in-app for traceable demo playback.

## 6. Scalability Considerations
This prototype can evolve toward production by:
- replacing SQLite with PostgreSQL and adding migrations
- introducing background jobs for scheduled scoring and summaries
- adding role-based access and audit policies
- replacing heuristic NL routing with constrained LLM SQL planning and policy validation
- versioning model artifacts and adding monitoring for drift/performance
- adding API/service boundaries between UI, feature pipelines, and inference layer

## 7. Risk and Safety Controls
- Only read-oriented/allowlisted analytical queries are generated by NL routing.
- Follow-up context is constrained to explicit slots (region, plan, intent).
- Model outputs are presented with explanatory fields to avoid black-box-only UX.
- Synthetic data avoids any real customer information.

## 8. Testing and Validation
The project includes tests that validate:
- synthetic volume and integrity
- health score boundaries and component schema
- churn model metric outputs
- follow-up context behavior in NL queries

The end-to-end demo runbook is provided in the README.
